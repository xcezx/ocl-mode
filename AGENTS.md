# Repository Guidelines

## Project Structure & Module Organization

This repository contains an Emacs major mode for Octopus Configuration Language.
The main implementation lives in `ocl-mode.el`. Tests are under `test/` and are
split by behavior: command navigation, syntax highlighting, indentation, and
shared helpers. CI configuration is in `.github/workflows/test.yml`; dependency
automation is in `.github/dependabot.yml`.

## Build, Test, and Development Commands

- `make`: byte-compiles the package and tests, runs the ERT suite, runs
  `package-lint`, then removes generated `.elc` files.
- `make test`: runs only the ERT tests from `test/`.
- `make package-lint`: checks package metadata and common Emacs Lisp packaging
  issues.
- `make clean`: removes generated byte-compiled files.

Set `EMACS=/path/to/emacs make test` to test against a specific Emacs binary.

## Coding Style & Naming Conventions

Use Emacs Lisp conventions already present in `ocl-mode.el`: lexical binding,
two-space indentation for Lisp forms, clear docstrings for public functions, and
private helpers prefixed with `ocl--`. Public commands and customization
variables use the `ocl-` prefix. Test helpers use `ocl-test--` for private test
utilities.

Keep OCL behavior faithful to the reference format. Do not add comment syntax,
string interpolation, expressions, function calls, or exponential number parsing
unless the upstream OCL format supports them.

## Testing Guidelines

Tests use Emacs ERT. Add or update tests in the most specific file:
`test-command.el` for navigation commands, `test-highlighting.el` for font-lock,
and `test-indentation.el` for indentation. Name tests with
`ocl-test-<behavior>`, and use `with-ocl-temp-buffer` from `test-helper.el` for
mode setup.

Run `make test` for focused checks and `make` before submitting changes.

## Commit & Pull Request Guidelines

Recent history uses short imperative subjects, optionally with a scope such as
`ci: pin GitHub Actions to full-length commit SHAs`. Keep commits focused and
describe observable behavior changes.

Pull requests should include a concise summary, test results such as `make`, and
links to relevant issues. Include screenshots only when UI behavior in Emacs is
material to the change.
